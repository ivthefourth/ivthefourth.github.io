General Front End

  • Separate HTML, CSS, and JS

    Unless there is good reason to do otherwise, HTML, CSS, and JS should be separated, with JS and CSS files linked to from the HTML. This keeps content, style, and functionality separated and makes for easier editing of these components.

  • Progressive Enhancement

    The core content and core functions of a website should work well on any device capable of browsing the web. This is ideally achieved by starting development with the basic content, and then adding more advanced functionality and design on top of that. This way, on more powerful devices or on browsers with more functionality, the user experience is enhanced, and the necessary fallbacks are built-in.

    If possible, javascript should only enhance the functionality. If it is absolutely necessary to the function of an application, there should at least be some explanation to the user using a <noscript> tag.

    • http://sixrevisions.com/web-development/progressive-enhancement/

  • Content First

    The content of a website should be created first, and then the design built around that content. Otherwise you may likely run into problems where your content doesn't fit within the design well or you have to create unnecessary "filler" content on some pages so that the design doesn't feel empty or unbalanced.

    This also relates closely to progressive enhancement. The content and its basic structure are the foundation of your site, and the design and additional functionality are built upon that foundation.

    • http://alistapart.com/article/content-modelling-a-master-skill

  • Responsive Design

    Unless you plan to have a separate site for mobile, it's best to make the design responsive. Many of the best practices on this page relate directly to responsive design.

    • @viewport{ width: device-width; } @-ms-viewport{ width: device-width; }
    • <meta name="viewport" content="width=device-width, initial-scale=1.0">

HTML

  • Semantic Elements & Structure

    Use semantic elements to structure your content appropriately. This is beneficial to both SEO and accessibility. For now, set the sectioning elements to display: block in CSS for legacy browsers

    Use header tags appropriately. h1 -> h2 -> h3, etc., with preferably just one h1 tag per page.

    • http://docs.webplatform.org/wiki/guides/lesser-known_semantic_elements

  • Style

    In most cases inline styles should not be used; styles should all be contained in linked style sheets (CSS) to keep design separate from content and make editing either more simple. HTML is for document outline, semantics, and content.

  • Responsive Images

    Use the <picture> element for art direction. Use a single src attribute for the fallback image; keep this relatively small when possible.

    • https://developer.mozilla.org/en-US/docs/Web/HTML/Element/picture
    • http://www.html5rocks.com/en/tutorials/responsive/picture-element/
    • https://webkit.org/demos/srcset/

  • Tables

    Consider your data and whether it is possible to display it as a list or description list rather than a table. Tables are very tricky on mobile, because they tend to take up so much horizontal space. If the data must be presented in a table, there are ways to style the table to make it display better on mobile.

    • http://blog.easy-designs.net/archives/responsive-tables/
    • https://css-tricks.com/examples/ResponsiveTables/responsive.php
    • http://dbushell.com/2016/03/04/css-only-responsive-tables/

  • Forms

    Focus on the content of the form fist. What information do you need from the user? Having the controls in a list can be effective in some cases. Make sure to use the label element and the for attribute as well as the aria-labelledby attribute. Use aria landmark roles role="form" and role="search" appropriately.

    • https://www.w3.org/WAI/intro/aria
    • https://www.smashingmagazine.com/2012/11/ux-design-qa-with-christian-holst/
    • http://diveintohtml5.info/forms.html

CSS

  • Maximum 3 Files Per Page

    Loading additional files requires more calls to the server and increases page load times. Therefore it is a good idea to load as few files as possible on each page, especially for mobile.

    CSS can be written in as many files as you wish, but at deployment these should all be combined and minimized into the maximum of three files per page.

    • https://css-tricks.com/one-two-three/

  • Layers of CSS

    It's a good idea to rely on CSS2 for the main structures of your site and enhance this with CSS3 (or browser prefixed styles). This way you don't have to worry about fallbacks for browsers that don't support these styles, as the fallbacks are built in already.

  • Layout

    It's best to start with the single-column (mobile) layout and then build the other layouts using min-width media queries with em sizing that are related to the how content of the site looks at different widths (rather than attempting to use device-specific media queries).

    Keep the single column layout relatively light and simple so that the pages load faster for mobile. Be sure that site content is visible on smaller screens before scrolling (no big headers), and only make users scroll in one direction to avoid confusion.

  • CSS Code

    Group CSS by selectors and related selectors (not by media queries). The result will be blocks with all CSS for a certain element on consecutive lines in the stylesheet, and related classes, etc., will be close in proximity for better readability and editing.

  • Responsive Typography

    the optimal number of characters on one line in a paragraph is 45- 75.

Front End Checklist

  • Define language in html tag: lang="en"
  • Use validators to check your code
  • Check document outline with HTML5 Outliner
  • The preferred order in the head: (1) charset, (2) title, (3) description, (4) viewport, (5) CSS.
  • Don't forget to add the charset meta tag for HTML5: <meta charset="UTF-8">
  • Declare the viewport using both CSS and HTML, for the future.
  • Make sure no horizontal scrolling is needed, even at the smallest viewports
  • Keep the head section clean and without redundancies
  • Write your HTML5 code with good syntax: lower case for element names and attributes, enclose attribute values in double quotation marks and close the elements;
  • Don't use divs you don't really need
  • Do not use text, images, or HTML to create white space for layout purposes: remember layout needs to be done in CSS
  • Avoid inline styles in HTML
  • Use HTML5, CSS3, and javascript according the approach of progressive enhancement
  • Place scripts in an external file for ease of maintenance and re-use
  • When sensible, load javascript at the end of the body element.
  • Use HTML5 structural semantical elements like header, nav, main, section, article, aside, footer etc., when appropriate
  • Avoid untitled sections (check the document outlines in the browser!) and use headings appropriately
  • Make use of the landmark role attribute to benefit screen readers
  • Use a list for the menus
  • Use label elements for forms
  • Only use em and strong semantically, not for style
  • Use relative sizes for text and layout: % or em
  • use each of a:hover, a:focus, a:active for accessibility
  • If text is uppercase for style purposes, make sure to do this with CSS and type the text normally in HTML
  • Be aware that display: none will also hide the content for screen readers.
